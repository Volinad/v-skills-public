# Spec Compiler — Learnings

A learning here exists to change what a FUTURE run does. This file holds only patterns that are **novel** (not already in SKILL.md or the holdout files) and **actionable** (a concrete check). Read by every Auditor at the start of a run.

This file starts EMPTY in a fresh installation and accumulates per-installation: the Phase 7 learning-collector appends an entry only when a run surfaces something genuinely new (see Phase 7 in SKILL.md for the bar). Most runs of a mature skill add nothing — that is the healthy outcome. Reconfirmations of patterns already in the gate/holdout/SKILL are NOT recorded; they live in each run's `VALIDATION-REPORT.md`. Once an entry's substance is implemented anywhere a run reads it, it collapses to a one-line pointer in **Archived**. This file is a live watch list, not a run log.

## Statuses

| Status | Meaning |
|--------|---------|
| NOTED | Active insight, not yet promoted — kept here until a 2nd occurrence promotes it to holdout |
| MATERIALIZED | Implemented as a holdout scenario/antipattern — moved to Archived |
| RESOLVED | Implemented in SKILL.md / phase logic — moved to Archived |
| REJECTED | Considered and intentionally not implemented |

## What belongs here (the bar for a new entry)

Apply two tests before writing an entry — write it only if BOTH pass:
- **Novel** — absent from SKILL.md, holdout, and existing entries. A fresh instance of an existing scenario/antipattern/gate is not novel; record it in the run's VALIDATION-REPORT, not here.
- **Actionable** — it changes a future run (a new check, method weighting, or document-class posture), statable in one sentence.

Keep each entry to its reusable core: the pattern + the check it implies, a few sentences, naming at most the ONE closest entry it refines. No "Nth confirmation" counters, no run narratives.
